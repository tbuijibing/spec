# 多客户端规范

> 目标：为 Web / Mobile / MiniApp / Desktop 等多客户端项目提供统一的 specs 组织方式。

---

## 一、设计原则

- **共享后端契约，分离客户端规格**。
- **客户端差异只落到差异层，不复制共享内容**。
- **版本管理优先按模块，再按客户端细分**。

---

## 二、结构策略

### 2.1 单客户端

使用当前扁平结构即可：
- `SPEC-XXX-3-UI-UX设计文档.md`
- `SPEC-XXX-4-任务文档.md`
- `SPEC-XXX-5-验收文档.md`

### 2.2 双客户端

建议通过文件后缀区分：
- `SPEC-XXX-3-UI-UX-Web.md`
- `SPEC-XXX-3-UI-UX-Mobile.md`

### 2.3 三个及以上客户端

建议升级为：

```text
SPEC-XXX-模块名/
├── shared/
│   ├── 需求文档
│   ├── 技术设计文档
│   ├── 数据库文档
│   └── 可追溯性矩阵
└── clients/
    ├── web/
    ├── mobile/
    ├── miniapp/
    └── desktop/
```

---

## 三、职责划分

| 层 | 放什么 |
|----|--------|
| `shared/` | 后端、API、数据库、共享业务规则 |
| `clients/web/` | Web UI/任务/验收/资产 |
| `clients/mobile/` | 移动端 UI/任务/验收/资产 |
| `clients/miniapp/` | 小程序 UI/任务/验收/资产 |

---

## 四、版本归档

多客户端版本归档应支持：
- 同版本不同客户端的测试结果拆分
- 同版本共享改动与客户端差异同时可见

示例：

```text
versions/v1.1/
├── 变更记录.md
└── test-report/
    ├── web/
    ├── mobile/
    └── miniapp/
```

---

## 五、适用时机

| 情况 | 是否升级多客户端结构 |
|------|----------------------|
| 只有后台 Web | 否 |
| Web + 移动端 | 视复杂度而定 |
| Web + 移动端 + 小程序 | 是 |
| 多品牌 / 多终端并行 | 是 |

---

## 六、禁止项

- 禁止把所有客户端 UI 差异强塞在同一份 UI 文档里。
- 禁止共享逻辑和客户端逻辑混写，导致无法维护。
- 禁止多客户端项目仍沿用单客户端目录造成失控。
