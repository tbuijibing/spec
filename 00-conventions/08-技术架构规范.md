# 技术架构规范

> 来源：`plans/specs/需求规范-02-技术架构.md`

---

## 一、分层架构铁律

统一采用：

```text
Controller → Service 接口 → ServiceImpl → Mapper → Entity
```

| 层 | 职责 | 禁止事项 |
|---|------|----------|
| Controller | 入参接收、权限校验、返回封装 | 写复杂业务逻辑 |
| Service | 业务编排、事务边界 | 直接拼接 SQL |
| Mapper | 数据访问、复杂查询 | 承担业务决策 |
| Entity | 表结构映射 | 承担业务行为 |

---

## 二、微服务边界

| 服务 | 负责模块 |
|------|----------|
| `globehive-base` | 用户画像、审计、租户、权限、基础配置 |
| `globehive-chat` | 会话、消息、渠道接入 |
| `globehive-speechcraft` | 话术、A/B 测试 |
| `globehive-marketing` | 营销任务、推送、活动、策略 |
| `globehive-insight` | 洞察、热点、推荐、系统进化 |
| `globehive-explore` | 用户探索、记忆、偏好、行为 |
| `globehive-aiengine` | Agent、语音、视频、MCP 能力承接 |
| `globehive-group` | 群聊、群成员、群运营 |

---

## 三、多租户约束

- 默认采用 BladeX `tenant_id` 表级隔离。
- 所有业务主表必须具备 `tenant_id`。
- 任何查询默认只能返回当前租户数据。
- 超级管理员跨租户能力必须显式授权，不允许隐式穿透。

---

## 四、横向能力架构

### 4.1 Agent

- Agent 是横向能力，不是随意落到任意业务模块中的“万能对象”。
- Agent 的配置、生命周期、训练、测试、上线必须有明确状态流转。
- 业务模块只能调用 Agent 能力，不应复制一套 Agent 逻辑。

### 4.2 MCP

- MCP Host 由 GlobeHive AI 平台承担。
- MCP Client 统一接入层，不允许每个模块私自实现一套协议适配。
- MCP Server 的 Tool / Resource / Prompt 设计必须保持可追溯。

---

## 五、架构设计要求

| 项目 | 要求 |
|------|------|
| 模块拆分 | 一个模块只负责一个清晰业务能力 |
| 跨模块调用 | 优先通过明确 API / Service 边界，不绕过边界直接访问内部实现 |
| 事务范围 | 只在 Service 层声明事务 |
| 配置管理 | 统一配置中心，不允许散落硬编码 |
| 可追溯性 | 需求 → API → 表 → 代码 → 测试必须可映射 |

---

## 六、架构文档要求

每个模块的 `SPEC-XXX-2-技术设计文档.md` 至少应包含：
- 分层结构图
- 核心数据流
- API 设计
- 表结构依赖
- 外部依赖
- 事务边界
- 异常边界

---

## 七、禁止项

- 禁止跨模块复制同类能力实现。
- 禁止在 Controller 中直接操作 Mapper。
- 禁止没有边界说明的“公共工具类”无序扩散。
- 禁止模块职责混杂，导致 README 与代码边界不一致。
