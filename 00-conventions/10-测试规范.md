# 测试规范

> 来源：`plans/测试策略文档.md`

---

## 一、测试目标

- 功能符合需求规格。
- 接口、前端、数据库行为一致。
- 每次迭代有回归依据。
- 验收结果能沉淀到版本归档中。

---

## 二、测试层次

| 层次 | 目标占比 | 说明 |
|------|----------|------|
| 单元测试 | 60% | 函数、服务、核心业务逻辑 |
| 集成测试 | 30% | API、数据库、缓存、消息队列协作 |
| E2E / 验收测试 | 10% | 浏览器真实操作、完整业务闭环 |

---

## 三、每模块必测项

| 类别 | 要求 |
|------|------|
| API 测试 | 正常、边界、异常、未授权、越权 |
| 功能测试 | 主流程、关键分支、回退路径 |
| 数据测试 | 建表、写入、更新、软删除、索引命中 |
| 前端测试 | 页面渲染、交互、状态提示、空状态 |
| 安全测试 | 租户隔离、权限、输入攻击 |
| 回归测试 | 修改点关联功能回归 |

---

## 四、验收要求

每个模块上线前必须完成：

1. 浏览器全流程真实操作
2. 截图存档
3. `SPEC-XXX-9-验收报告.md` 填写完成
4. 版本目录 `versions/vX.Y/test-report/` 归档完成

---

## 五、覆盖率目标

| 范围 | 目标 |
|------|------|
| 核心服务层 | ≥80% |
| 高风险模块 | ≥85% |
| 工具类 / 适配层 | 视价值评估，但不得零测试 |

---

## 六、与模块文档关系

| 文档 | 作用 |
|------|------|
| `SPEC-XXX-5-验收文档.md` | 定义测试范围与用例 |
| `SPEC-XXX-9-验收报告.md` | 记录验收结果与结论 |
| `versions/vX.Y/test-report/` | 保存该版本所有测试证据 |

---

## 七、禁止项

- 禁止“代码写完但未验收”就标记完成。
- 禁止只测接口，不测真实页面流程。
- 禁止修改后不做回归。
- 禁止没有测试证据就写“通过”。

---

## 八、代码审查规范

### 8.1 审查时机

- 每个模块功能开发完成后，自测通过，提交前。
- Hotfix 修复完成后，合并前。

### 8.2 审查清单

| 维度 | 检查内容 |
|------|----------|
| 命名 | 是否符合 `03-命名规范.md` |
| 数据库 | 是否有无索引查询、SELECT *、字符串拼 SQL |
| 安全 | 是否做了租户隔离、权限校验 |
| 异常处理 | 是否有裸 `catch (Exception e) {}` 吞异常 |
| AI 安全 | 用户输入是否直接拼进 Prompt |
| 日志 | 是否有敏感信息明文输出 |

### 8.3 审查方式

单人项目可自审，自审时**必须在提交 PR/Merge 前间隔至少 30 分钟**（避免刚写完就审查的盲区）。
多人项目必须由另一名成员审查。

### 8.4 审查结果记录

审查意见及修改结果记录在 Git Commit Message 或 PR 描述中，不需要额外文档。

## 九、测试数据管理

### 9.1 数据隔离原则

- 所有测试数据必须归属于**专用测试租户**（tenant_id = `test001`）。
- 禁止在 test 环境使用生产数据的真实副本（涉及个人信息）。

### 9.2 测试数据准备

| 数据类型 | 准备方式 |
|----------|----------|
| 基础数据（用户、租户） | 固定 SQL 脚本，存放于 `sql/test-data.sql` |
| 业务测试数据 | 通过接口或页面操作创建，验收完成后截图存档 |
| AI 测试样本 | 存放于 `prompts/test-cases/`，覆盖正常、边界、攻击三类 |

### 9.3 数据清理规范

- 每次验收结束后，清理本次创建的测试数据（保留基础数据）。
- 清理脚本统一命名：`sql/cleanup-test-data.sql`。
- 禁止在 test 环境遗留大量无用数据（影响下一轮测试的准确性）。

### 9.4 禁止项

- 禁止用真实手机号、邮箱、身份证作为测试数据。
- 禁止测试完成后不清理直接关闭任务。
- 禁止 prod 环境执行测试操作。
