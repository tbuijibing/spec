# 国际化规范

> 来源：`plans/specs/需求规范-11-多语言i18n体系.md`

---

## 一、三层翻译架构

| 层 | 内容 | 实现方式 |
|----|------|----------|
| UI 静态文案 | 菜单、按钮、提示、状态 | 前端 i18n 资源 |
| 业务内容翻译 | Agent 名称、话术、营销文案、商品描述 | `*_translate` 表 |
| 实时消息翻译 | 用户消息、客服回复实时互译 | 翻译 API / 网关 |

---

## 二、语言代码

统一采用 **BCP 47**，如：
- `zh-CN`
- `zh-TW`
- `en-US`
- `ja-JP`
- `ko-KR`
- `ar-SA`

---

## 三、规范要求

| 项目 | 规范 |
|------|------|
| UI 文案 | 禁止硬编码，必须走 i18n key |
| 业务内容 | 有多语言需求的业务表必须配套翻译表 |
| 默认语言 | 项目默认 `zh-CN`，租户可配置默认语言 |
| 用户偏好 | 优先读取用户首选语言 |
| RTL | 阿拉伯语等 RTL 语言需额外适配布局 |

---

## 四、翻译表要求

- 命名：`{原表名}_translate`
- 唯一键：`({业务id}, lang)`
- 语言字段值必须用标准语言代码
- 所有翻译字段应与原业务字段语义一致

---

## 五、模块文档要求

涉及国际化的模块，至少要在：
- 需求文档中说明语言范围
- 数据库文档中说明翻译表
- 验收文档中说明多语言测试场景

---

## 六、禁止项

- 禁止把面向用户的多语言文本写死在代码中。
- 禁止使用非标准语言代码。
- 禁止只翻译 UI，不翻译业务内容。
