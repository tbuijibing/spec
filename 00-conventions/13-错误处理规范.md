# 错误处理规范

> 目标：统一前后端错误边界、错误码、异常分类与用户提示方式。

---

## 一、异常分类

| 类型 | 场景 | 建议返回 |
|------|------|----------|
| 参数异常 | 缺参、格式错误、非法值 | 400 |
| 业务异常 | 状态不允许、规则冲突 | 400/409 |
| 未认证 | Token 失效、缺失 | 401 |
| 未授权 | 权限不足、跨租户访问 | 403 |
| 资源不存在 | 数据不存在 | 404 |
| 系统异常 | 依赖故障、未知异常 | 500 |

---

## 二、后端处理边界

- Controller 只做参数校验和返回包装。
- Service 抛出业务语义明确的异常。
- 全局异常处理负责把异常映射为统一响应。
- 记录错误日志时不得泄漏敏感信息。

---

## 三、前端处理边界

- 表单错误：贴近字段展示。
- 业务错误：展示明确可理解的信息。
- 系统错误：展示统一兜底提示，并保留追踪信息。
- 不允许把后端堆栈直接展示给用户。

---

## 四、错误码建议

| 范围 | 说明 |
|------|------|
| `400xx` | 参数 / 业务异常 |
| `401xx` | 认证失败 |
| `403xx` | 权限问题 |
| `404xx` | 资源不存在 |
| `500xx` | 系统异常 |

---

## 五、文档要求

模块 `SPEC-XXX-5` 与 `SPEC-XXX-7` 应覆盖：
- 常见错误场景
- 错误码示例
- 排查步骤
- 用户提示方式

---

## 六、禁止项

- 禁止吞掉异常后静默失败。
- 禁止无上下文的“操作失败”。
- 禁止把兜底逻辑写成大面积无差别 try-catch。
- 禁止错误与日志不留痕。
