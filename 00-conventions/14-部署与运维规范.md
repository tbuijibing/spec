# 部署与运维规范

> 来源：`plans/监控与运维方案.md`、`plans/specs/需求规范-04-最佳实践与实施计划.md`

---

## 一、环境分层

| 环境 | 用途 |
|------|------|
| dev | 本地开发与联调 |
| test | 集成测试、验收演练 |
| prod | 正式生产 |

---

## 二、发布前检查

- 文档齐全（至少模块 5/7/9 文档可用）
- 数据库变更已评审
- 回滚方案明确
- 核心接口已测试通过
- 关键监控与告警已配置

---

## 三、发布要求

| 项目 | 要求 |
|------|------|
| 发布窗口 | 选择低风险时段 |
| 变更说明 | 必须有清晰版本说明 |
| 灰度 / 分批 | 高风险模块优先灰度 |
| 回滚预案 | 数据、服务、配置三类回滚都要可执行 |

---

## 四、备份与恢复

- 数据库备份必须有频率、保留策略、恢复演练。
- 配置文件与规范文档必须版本化管理。
- 恢复步骤要写入运维文档，不允许只存在口头经验。

---

## 五、上线后观察

上线后至少观察：
- 错误率
- 响应时间
- 慢查询
- 关键业务成功率
- AI / 第三方接口失败率

---

## 六、模块文档要求

版本目录 `versions/vX.Y/` 至少应有：
- `变更记录.md`
- `test-report/`
- 设计快照（如有 UI 变更）

---

## 七、热修复（Hotfix）流程

> 适用场景：生产环境出现紧急 Bug，需要在当前迭代串行链之外插入紧急修复。

### 7.1 触发条件

满足以下任一条件即可启动 Hotfix：
- 生产核心功能不可用（P0）
- 数据安全或权限漏洞被发现
- 影响超过 10% 用户的功能异常（P1）

### 7.2 Hotfix 执行步骤

```
1. 冻结当前进行中模块的开发（保存现场，不提交）
2. 从 main/master 创建 hotfix 分支：hotfix/YYYY-MM-DD-{描述}
3. 最小化修复：只改 Bug，不重构，不捎带功能
4. 本地验证 + test 环境验证
5. 合并回 main 并部署生产
6. 将 hotfix 变更同步到当前开发分支（cherry-pick 或 merge）
7. 记录到 20-踩坑记录.md（现象 → 根因 → 解决 → 预防）
8. 恢复被冻结的模块开发
```

### 7.3 Hotfix 禁止项

- 禁止在 Hotfix 中做功能扩展。
- 禁止跳过 test 环境直接上 prod。
- 禁止 Hotfix 结束后忘记同步回开发分支（否则修复会被覆盖）。

### 7.4 与串行开发规则的关系

Hotfix 是**串行规则的例外通道**，使用后必须恢复串行。Hotfix 的存在不允许演变为长期并行开发的借口。
