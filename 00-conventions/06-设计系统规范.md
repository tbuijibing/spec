# 设计系统规范

> 来源：`project-specs/02-frontend/DESIGN-001-UI设计规范.md`  
> 目标：统一 {项目名} 的界面视觉、组件行为与可访问性标准。

---

## 一、适用范围

| 范围 | 说明 |
|------|------|
| Web 管理后台 | 当前主客户端，优先级 P0 |
| 后续移动端 / 小程序 | 视觉语言需继承本规范，再做客户端适配 |
| 模块 UI/UX 文档 | `SPEC-XXX-3-UI-UX设计文档.md` 必须引用本规范 |

---

## 二、核心原则

- **一致性优先**：同类信息用同类组件，不为局部需求随意造型。
- **语义化 Token**：颜色、间距、圆角、字体必须用设计 Token，不硬编码视觉值。
- **组件优先**：先复用组件，再写页面；先复用模式，再写交互。
- **可访问性优先**：键盘导航、焦点、ARIA、对比度是必选项。
- **中英混排友好**：兼容 {项目名} 的国际化场景。

---

## 三、设计体系

| 维度 | 规范 |
|------|------|
| 设计源 | Figma 为主，Pencil 可作为快照/备份 |
| 组件体系 | 沿用项目统一组件规范，不允许模块自行发明基础组件 |
| 主题 | Light / Dark 双主题，颜色通过变量控制 |
| 图标 | 统一线性风格，语义明确，避免混用多套图标 |
| 字体 | 英文 Inter / 中文 Noto Sans SC（或项目统一中文字体） |

---

## 四、设计 Token 约束

### 4.1 颜色

| Token 类型 | 说明 |
|-----------|------|
| `primary` | 品牌主色、主按钮、关键链接 |
| `secondary` | 次级操作、次强调信息 |
| `muted` | 弱化文本、辅助信息 |
| `destructive` | 删除、失败、危险操作 |
| `border` / `input` / `ring` | 输入框、边框、焦点状态 |

### 4.2 间距

- 采用 **4px 基准**。
- 常用间距：`4 / 8 / 12 / 16 / 24 / 32`。
- 禁止随意使用不成体系的零散像素值。

### 4.3 圆角与阴影

| 项目 | 规范 |
|------|------|
| 小组件 | 6~8px |
| 卡片 / 弹窗 | 8~12px |
| 阴影 | 轻量、分层明确，不做重投影堆叠 |

---

## 五、组件规范

| 组件 | 要求 |
|------|------|
| Button | 主 / 次 / 危险 / 文本态语义清晰 |
| Input / Select / DatePicker | 校验、禁用、错误、聚焦状态必须完整 |
| Table | 排序、筛选、分页、空状态、加载状态必须统一 |
| Dialog / Drawer | 标题区、内容区、操作区布局固定 |
| Form | 必填项、错误提示、字段说明位置统一 |
| Tag / Badge | 仅用于状态、标签、轻量分类，不替代表单输入 |

---

## 六、页面级规范

| 页面区域 | 标准结构 |
|---------|----------|
| 列表页 | 顶部标题 + 工具栏 + 搜索区 + 表格 + 分页 |
| 详情页 | 基础信息卡 + 子模块分区 + 时间线/日志 |
| 表单页 | 分组标题 + 字段区 + 底部操作区 |
| 运营页 | 筛选条件 + 指标卡 + 图表 + 明细表 |

---

## 七、可访问性规范

- 文本与背景对比度必须可读。
- 所有关键交互可通过键盘完成。
- 图标按钮必须有 Tooltip 或 ARIA Label。
- 表单错误必须有明确文本说明，不能只靠颜色区分。
- 暗黑模式下不能出现信息缺失或状态不可辨识。
