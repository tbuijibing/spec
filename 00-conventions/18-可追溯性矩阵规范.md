# 可追溯性矩阵规范

> 来源：`plans/可追溯性矩阵规范.md`、`plans/specs/{项目名}_可追溯性矩阵.md`  
> 目标：统一 {项目名}项目中需求、设计、路由、API、代码、数据表、测试之间的映射规则，确保模块实现可追踪、可验收、可排查。

---

## 一、适用范围

| 范围 | 要求 |
|------|------|
| 每个模块 | 必须提供 `SPEC-XXX-8-可追溯性矩阵.md` |
| 需求文档 | 需求项必须能映射到 API / 测试 |
| UI/UX 文档 | 页面与设计图必须能映射到路由 / 组件 |
| 技术设计文档 | API、表、代码实现必须可追踪 |
| 验收文档与验收报告 | 测试结论必须能回到需求与实现 |

---

## 二、核心原则

- **一条需求，必须能找到对应实现与验证证据**。
- **一条 API，必须知道它服务于哪个需求**。
- **一个页面，必须知道对应哪张设计图、哪个路由、哪些接口**。
- **一个测试结论，必须能回溯到需求、代码、接口、页面**。
- **矩阵是证据索引，不是正文替代品**。

---

## 三、编号体系建议

### 3.1 需求编号

格式：`REQ-{模块代码}-{流水号}`

示例：
- `REQ-USR-001`
- `REQ-CHT-003`
- `REQ-MKT-012`

### 3.2 设计图编号

格式：`UIX-{模块代码}-{页面类型}-{流水号}`

| 页面类型 | 含义 |
|----------|------|
| `LIST` | 列表页 |
| `FORM` | 表单页 |
| `DETAIL` | 详情页 |
| `DASH` | 仪表板 |
| `DIALOG` | 弹窗 |
| `DRAWER` | 抽屉 |

示例：
- `UIX-USR-LIST-001`
- `UIX-USR-FORM-001`
- `UIX-USR-DIALOG-001`

### 3.3 路由编号

格式：`RTE-{端代码}-{模块代码}-{页面类型}-{流水号}`

| 端代码 | 含义 |
|--------|------|
| `WEB` | Web 后台 |
| `MOB` | H5 / Mobile |
| `MINI` | 小程序 |
| `IOS` | iOS |
| `AND` | Android |
| `DSK` | Desktop |

示例：
- `RTE-WEB-USR-LIST-001`
- `RTE-MOB-USR-DETAIL-001`

### 3.4 接口编号

格式：`API-{模块代码}-{操作类型}-{流水号}`

| 操作类型 | 含义 |
|----------|------|
| `QRY` | 查询 |
| `CMD` | 创建 |
| `UPT` | 更新 |
| `DEL` | 删除 |
| `ACT` | 动作 / 特殊操作 |

示例：
- `API-USR-QRY-001`
- `API-USR-UPT-001`
- `API-USR-ACT-001`

### 3.5 测试编号

格式：`TST-{测试类型}-{模块代码}-{流水号}`

| 测试类型 | 含义 |
|----------|------|
| `UNIT` | 单元测试 |
| `INTG` | 集成测试 |
| `API` | 接口测试 |
| `E2E` | 端到端测试 |
| `UAT` | 验收测试 |
| `SEC` | 安全测试 |
| `PERF` | 性能测试 |

示例：
- `TST-API-USR-001`
- `TST-E2E-USR-001`
- `TST-SEC-USR-001`

## 四、矩阵最小结构

每个模块的 `SPEC-XXX-8-可追溯性矩阵.md` 至少包含以下 6 类映射：

### 4.1 需求 → 设计 / 路由 / API / 测试

| 需求编号 | 需求名称 | 设计图 | 路由 | API | 测试 |
|----------|----------|--------|------|-----|------|
| `REQ-USR-001` | 用户画像列表查询 | `UIX-USR-LIST-001` | `RTE-WEB-USR-LIST-001` | `API-USR-QRY-001` | `TST-E2E-USR-001` |

### 4.2 API → 代码文件

| API 编号 | URL / 方法 | Controller | Service | Mapper | 备注 |
|----------|------------|------------|---------|--------|------|
| `API-USR-QRY-001` | `GET /user-profiles/page` | `AiUserProfileController.page()` | `pageProfile()` | `AiUserProfileMapper.xml` | 分页查询 |

### 4.3 页面 / 设计图 → 组件 / 路由 / API

| 设计图 | 页面/组件 | 路由 | 关联 API |
|--------|-----------|------|----------|
| `UIX-USR-LIST-001` | `userProfile.vue` | `/{项目名}/base/userProfile` | `API-USR-QRY-001` |

### 4.4 数据表 → 代码映射

| 表名 | Entity | Mapper | Service | 说明 |
|------|--------|--------|---------|------|
| `ai_user_profile` | `AiUserProfile` | `AiUserProfileMapper` | `AiUserProfileServiceImpl` | 用户画像主表 |

### 4.5 源码文件清单

| 文件 | 包路径 / 路径 | 说明 |
|------|---------------|------|
| `AiUserProfileController.java` | `com.{项目名}.base.controller` | 控制器 |
| `userProfile.vue` | `src/views/{项目名}/base` | 页面组件 |

### 4.6 测试证据映射

| 测试编号 | 测试类型 | 覆盖需求 / API | 证据位置 |
|----------|----------|----------------|----------|
| `TST-E2E-USR-001` | E2E | `REQ-USR-001` | `versions/v1.0/test-report/...` |

## 五、推荐文档结构

推荐 `SPEC-XXX-8-可追溯性矩阵.md` 使用以下章节：

1. **需求 → API / 设计映射**
2. **API → 代码文件映射**
3. **API → 测试用例映射**
4. **数据库表 → 代码映射**
5. **页面 / 路由 / 设计图映射**（如果该模块有 UI）
6. **源码文件清单**
7. **版本历史**

## 六、与其他文档的关系

| 文档 | 关系 |
|------|------|
| `SPEC-XXX-1-需求文档.md` | 提供需求来源 |
| `SPEC-XXX-2-技术设计文档.md` | 提供 API / 表 / 分层实现来源 |
| `SPEC-XXX-3-UI-UX设计文档.md` | 提供页面、组件、设计图来源 |
| `SPEC-XXX-5-验收文档.md` | 提供测试范围与用例来源 |
| `SPEC-XXX-9-验收报告.md` | 提供最终结论与结果证据 |

## 七、多客户端扩展规则

当模块涉及多个客户端时，矩阵需增加按客户端拆分的路由和页面映射列：

| 需求编号 | Web 路由 | Mobile 路由 | MiniApp 路由 | 共享 API |
|----------|----------|-------------|--------------|----------|
| `REQ-XXX-001` | `RTE-WEB-...` | `RTE-MOB-...` | `RTE-MINI-...` | `API-XXX-QRY-001` |

规则：
- 后端 API、数据库、共享业务规则通常放在共享层
- 页面路由、组件、设计图按客户端拆分
- 不允许多个客户端混写成无法区分的单表，除非模块只有单客户端

---

## 八、编写要求

- 矩阵表必须真实、可验证，禁止“为了完整而编造编号”。
- 若模块没有某类内容，可明确写 `—`，不能空着不解释。
- 所有路径、类名、方法名、接口 URL 必须与实际实现一致。
- 每次大版本迭代后，矩阵必须与模块版本归档同步更新。

---

## 九、禁止项

- 禁止只列需求，不列实现与测试。
- 禁止矩阵与真实文件路径不一致。
- 禁止矩阵长期不更新，变成摆设。
- 禁止把矩阵写成冗长正文，失去索引作用。
