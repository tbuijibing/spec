# 性能规范

> 来源：`plans/性能与扩展性设计文档.md`

---

## 一、核心性能指标

| 指标类别 | 指标 | 目标 |
|---------|------|------|
| 普通 API 响应 | P95 | ≤500ms |
| 数据库普通查询 | P95 | ≤100ms |
| 缓存查询 | P95 | ≤10ms |
| AI 回复 | P95 | ≤2s |
| 系统可用性 | 年度 | ≥99.9% |

---

## 二、阶段目标

| 阶段 | 并发用户 | API 响应 |
|------|----------|----------|
| MVP | ≥1,000 | ≤1s |
| 稳定版 | ≥5,000 | ≤500ms |
| 成熟版 | ≥10,000 | ≤500ms |

---

## 三、设计要求

| 项目 | 规范 |
|------|------|
| 数据库 | 建索引、禁用 `SELECT *`、控制慢查询 |
| 缓存 | 热点数据必须有缓存策略和失效策略 |
| 列表接口 | 必须分页，不允许无上限全量查询 |
| 批量任务 | 异步、分批处理、可观察 |
| AI 调用 | 要有超时、限流、降级策略 |

---

## 四、容量与扩展

- 模块设计时需说明是否需要读写分离、缓存预热、分库分表。
- 高吞吐模块需明确消息队列、缓存、存储压力评估。
- 性能目标应写入模块技术设计文档或验收文档。

---

## 五、测试要求

| 类型 | 要求 |
|------|------|
| 性能压测 | 关键 API 必须有压测结果 |
| 慢查询检查 | 上线前检查核心 SQL |
| 缓存命中率 | 核心缓存要定义目标命中率 |
| 回归验证 | 性能优化后要防止功能回退 |

---

## 六、禁止项

- 禁止无分页列表接口直接上线。
- 禁止无索引的高频查询。
- 禁止把性能问题留到生产后再定位。
- 禁止没有监控就宣称“性能达标”。
