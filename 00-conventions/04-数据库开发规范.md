# 数据库开发规范

> 数据库：MySQL 8.0 | 数据库名：`{项目名}` | 更新：2026-03-30

---

## 一、基础原则

| 规则 | 说明 |
|------|------|
| 禁止 PostgreSQL 语法 | 禁止 BIGSERIAL、TIMESTAMPTZ、JSONB；只用 MySQL 8.0 |
| 禁止 SELECT * | 所有查询必须明确列名 |
| 禁止硬编码文案 | 界面相关文案必须走 i18n |
| 禁止无索引查询 | `tenant_id`、`user_id`、`status`、时间范围必须有索引 |
| 软删除 | 统一用 `is_deleted=1`，不做物理删除 |

## 三、BladeX BaseEntity 标准字段

每张 {项目名} 业务表**必须**包含以下 9 个标准字段（顺序固定，置于业务字段之前）：

```sql
`id`          bigint       NOT NULL            COMMENT '主键（雪花算法，@TableId ASSIGN_ID）',
`tenant_id`   varchar(12)  DEFAULT '000000'    COMMENT '租户ID（关联 blade_tenant.tenant_id）',
`create_user` bigint       DEFAULT NULL        COMMENT '创建人（关联 blade_user.id）',
`create_dept` bigint       DEFAULT NULL        COMMENT '创建部门',
`create_time` datetime     DEFAULT NULL        COMMENT '创建时间',
`update_user` bigint       DEFAULT NULL        COMMENT '最后修改人',
`update_time` datetime     DEFAULT NULL        COMMENT '最后修改时间',
`status`      int          DEFAULT 1           COMMENT '状态：1=启用，0=禁用',
`is_deleted`  int          DEFAULT 0           COMMENT '软删除：0=正常，1=已删',
```

> **主键规则**：`bigint NOT NULL`，雪花算法，**非** `AUTO_INCREMENT`。Java 层用 `@TableId(type = IdType.ASSIGN_ID)`。

## 四、数据类型选择

| 用途 | MySQL 类型 | 说明 |
|------|-----------|------|
| 主键 | `bigint NOT NULL` | 雪花算法 |
| 外键（引用 blade_user.id） | `bigint` | |
| 租户 ID | `varchar(12)` | 关联 blade_tenant.tenant_id |
| 普通字符串 | `varchar(N)` | N 按业务设定，不超实际需求 |
| 状态枚举 | `tinyint` / `int` / `varchar(20)` | 视复杂度选择 |
| 时间 | `datetime` | 非 `timestamp`（时区问题） |
| 日期 | `date` | 仅需日期时用 |
| 布尔 | `tinyint(1)` | 0/1 |
| 金额（分） | `bigint` | 避免浮点精度问题 |
| JSON 数组/对象 | `json` | MyBatis 配 JacksonTypeHandler |
| 长文本 | `text` | 消息内容、备注等 |
| 超长文本 | `mediumtext` / `longtext` | 知识库内容等 |

## 五、索引规范

| 类型 | 命名前缀 | 示例 |
|------|----------|------|
| 唯一索引 | `uk_` | `uk_tenant_user` |
| 普通索引 | `idx_` | `idx_tenant_id`, `idx_create_time` |
| 全文索引 | `ft_` | `ft_message_content` |

**必建索引清单：**
- `idx_tenant_id`：所有业务表
- `uk_tenant_user`：需要租户内唯一的表（如 `ai_user_profile`）
- 涉及时间范围查询的字段：`create_time`
- 涉及状态过滤的高频查询：`status`, `is_deleted`

## 六、DDL 模板

```sql
CREATE TABLE `ai_{模块名}` (
  -- ── BladeX BaseEntity 标准字段 ──────────────────
  `id`          bigint       NOT NULL            COMMENT '主键（雪花算法）',
  `tenant_id`   varchar(12)  DEFAULT '000000'    COMMENT '租户ID',
  `create_user` bigint       DEFAULT NULL        COMMENT '创建人',
  `create_dept` bigint       DEFAULT NULL        COMMENT '创建部门',
  `create_time` datetime     DEFAULT NULL        COMMENT '创建时间',
  `update_user` bigint       DEFAULT NULL        COMMENT '最后修改人',
  `update_time` datetime     DEFAULT NULL        COMMENT '最后修改时间',
  `status`      int          DEFAULT 1           COMMENT '状态：1=启用，0=禁用',
  `is_deleted`  int          DEFAULT 0           COMMENT '软删除',
  -- ── 业务字段 ──────────────────────────────────
  `{字段名}`    {类型}       DEFAULT {默认值}    COMMENT '{说明}',
  -- ── 主键与索引 ────────────────────────────────
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_tenant_{字段}` (`tenant_id`, `{唯一字段}`),
  KEY `idx_tenant_id` (`tenant_id`),
  KEY `idx_create_time` (`create_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='{表说明}';
```

## 七、多语言翻译表规范

```sql
CREATE TABLE `ai_{原表名}_translate` (
  `id`          bigint    NOT NULL AUTO_INCREMENT COMMENT '主键',
  `{原表名}_id` bigint    NOT NULL               COMMENT '关联原表主键',
  `locale`      varchar(10) NOT NULL             COMMENT '语言代码（zh-CN/en-US）',
  `{字段名}`    varchar(N)  DEFAULT NULL         COMMENT '翻译内容',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_{原表名}_locale` (`{原表名}_id`, `locale`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci;
```

> 无外键约束，应用层维护关联关系。

## 八、数据库迁移策略

### 8.1 迁移方式

本项目采用**手动 SQL 脚本**管理数据库迁移（不引入 Flyway/Liquibase，保持与 BladeX 框架一致）。

### 8.2 脚本命名规范

```
SPEC-XXX-v{X.Y}-{描述}.sql
```

| 示例 | 说明 |
|------|------|
| `SPEC-101-v1.0-ddl.sql` | 初版建表脚本 |
| `SPEC-101-v1.1-add-column.sql` | 新增字段迁移脚本 |
| `SPEC-101-v1.1-index.sql` | 新增索引脚本 |

### 8.3 脚本存放位置

```
02-modules/SPEC-XXX-模块名/sql/
├── SPEC-XXX-ddl.sql          ← 当前全量建表脚本（始终保持最新）
└── migrations/
    ├── SPEC-XXX-v1.1-add-column.sql   ← 增量变更脚本
    └── SPEC-XXX-v1.2-add-index.sql
```

### 8.4 迁移执行规范

| 环境 | 执行方式 | 执行人 |
|------|----------|--------|
| dev | 开发自行执行 | 开发 |
| test | 发版前统一执行 | 测试/开发 |
| prod | 发版窗口执行，DBA 或负责人操作 | DBA / 项目负责人 |

### 8.5 迁移安全规则

- **禁止直接 DROP COLUMN**：先废弃（注释标记），下一版本再删除。
- **禁止无事务的批量 UPDATE**：大表数据迁移必须分批执行。
- **必须有回滚脚本**：每个迁移脚本旁附 `{同名}-rollback.sql`。
- **生产执行前必须在 test 环境验证通过**。
