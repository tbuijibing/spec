## 一、数据库

### DB-001 实际 DB 字段与 Spec 文档不一致

- **发现模块**：SPEC-101 用户画像
- **现象**：`ai_user_profile` 表缺少 `preferred_lang`、`status`、`is_deleted` 等字段，导致接口返回 Unknown column 错误。
- **根因**：历史迭代中 DDL 未同步更新，Entity 类字段与数据库实际字段存在 13 处差异。
- **解决方案**：执行 `SPEC-101-6-数据库文档.md §7.1` 中的修复 ALTER SQL。
- **预防措施**：每次迭代后必须执行 `SPEC-XXX-6-数据库文档.md` 中的差异清单校验；上线前用 `SHOW COLUMNS` 与 Entity 字段交叉比对。

---

### DB-002 BladeX BaseEntity 字段命名冲突

- **发现模块**：通用
- **现象**：自定义表中定义 `create_time` 字段后，MyBatis Plus 自动填充失败，且出现字段重复错误。
- **根因**：BladeX `BaseEntity` 已包含 `create_time`、`update_time`、`create_user`、`update_user`、`status`、`is_deleted`，业务表不能重复定义。
- **解决方案**：业务 Entity 继承 `BaseEntity`，删除重复字段定义。
- **预防措施**：建表前查阅 `specs/00-conventions/04-数据库开发规范.md § BladeX BaseEntity` 字段列表。

---

### DB-003 多租户场景下忘记加 tenant_id 索引

- **发现模块**：通用
- **现象**：分页查询 P95 > 2s，生产环境慢查询日志中出现全表扫描。
- **根因**：高频查询 WHERE 子句包含 `tenant_id + 业务字段`，但索引只建在业务字段上，未包含 `tenant_id` 作为最左前缀。
- **解决方案**：`ALTER TABLE xxx ADD INDEX idx_tenant_xxx (tenant_id, 业务字段);`
- **预防措施**：所有多租户业务表的复合索引，`tenant_id` 必须作为最左前缀。

## 二、接口与后端

### API-001 R\<T\> 包装层缺失导致前端解析失败

- **发现模块**：通用
- **现象**：前端调用接口拿到原始对象，`data.records` 为 undefined。
- **根因**：Controller 直接返回 `List<T>` 或 `Page<T>`，未用 `R.data(...)` 包装。
- **解决方案**：所有接口统一返回 `R<T>`，查询用 `R.data(result)`，操作用 `R.status(true/false)`。
- **预防措施**：参考 `specs/00-conventions/05-接口设计规范.md § 返回格式`。

---

### API-002 分页接口 current/size 参数未做最大值限制

- **发现模块**：SPEC-101、SPEC-102
- **现象**：传入 `size=10000` 导致单次查询拉取大量数据，内存 OOM。
- **根因**：未对 `size` 参数上限做校验，MyBatis Plus 直接透传。
- **解决方案**：在 BaseQuery 或接口层增加 `@Max(200)` 校验，超出时截断到 200。
- **预防措施**：所有分页接口在 VO/DTO 层加 JSR-303 校验注解。

---

### API-003 DELETE 接口误用路径参数导致批量删除失败

- **发现模块**：SPEC-101
- **现象**：批量逻辑删除走 `POST /remove`（ids 逗号分隔），但前端误调用 `DELETE /{id}` 逐条删除，产生 N+1 请求。
- **根因**：API 文档未明确标注批量删除入口，前端按 RESTful 惯例猜测接口。
- **解决方案**：在 `SPEC-XXX-2-技术设计文档.md` 中明确标注批量操作入口及参数格式。
- **预防措施**：非标准 RESTful 接口（如批量操作）必须在文档中加 ⚠️ 提示。

## 三、认证与权限

### AUTH-001 跨租户数据泄漏

- **发现模块**：通用
- **现象**：租户A管理员能查询到租户B的用户数据。
- **根因**：Service 层查询未加 `tenant_id` 过滤，依赖前端传参而非服务端强制隔离。
- **解决方案**：在 Service 层通过 `BladeX TenantContextHolder.getTenantId()` 获取当前租户 ID，强制注入查询条件。
- **预防措施**：所有业务查询 Wrapper 必须加 `.eq("tenant_id", tenantId)`；代码审查时优先检查此项。

---

### AUTH-002 Token 刷新后旧 Token 仍可使用

- **发现模块**：SPEC-115 成员角色权限
- **现象**：用户修改密码后，旧 Token 在过期前仍能正常调用接口。
- **根因**：JWT 无状态，服务端未维护 Token 黑名单。
- **解决方案**：密码变更后将旧 Token 的 JTI 写入 Redis 黑名单，网关拦截器校验黑名单。
- **预防措施**：涉及安全敏感操作（改密、踢人）必须同步使旧 Token 失效。

## 四、部署与环境

### DEPLOY-001 Nacos 配置未热更新导致服务使用旧配置

- **发现模块**：通用
- **现象**：修改 Nacos 中的配置后，服务仍使用旧值，重启后才生效。
- **根因**：配置类未加 `@RefreshScope` 注解，无法感知 Nacos 配置变更。
- **解决方案**：在需要热更新的 `@ConfigurationProperties` 类或 `@Value` 字段所在 Bean 上加 `@RefreshScope`。
- **预防措施**：所有从 Nacos 读取的配置 Bean 默认加 `@RefreshScope`。

---

### DEPLOY-002 Docker 镜像中文乱码

- **发现模块**：通用
- **现象**：容器日志中中文字符显示为乱码，数据库写入中文后查询为问号。
- **根因**：基础镜像未设置 `LANG=zh_CN.UTF-8`，JVM 未指定 `-Dfile.encoding=UTF-8`。
- **解决方案**：Dockerfile 中添加 `ENV LANG=zh_CN.UTF-8` 和 `ENV JAVA_OPTS="-Dfile.encoding=UTF-8"`。
- **预防措施**：使用统一的基础镜像 `globehive-base-jdk17`，已内置编码设置。

---

### DEPLOY-003 PowerShell 脚本中文路径编码失败

- **发现模块**：规范文档构建脚本
- **现象**：包含中文路径的 `.ps1` 脚本执行时报 `哈希文本中的键后面缺少"="运算符` 错误。
- **根因**：PowerShell 默认以 GBK 读取脚本文件，UTF-8 编码的中文字符解析失败。
- **解决方案**：改用 `Get-ChildItem` 动态获取目录路径，避免在脚本中硬编码中文路径；或在脚本首行加 BOM 标记（UTF-8 with BOM）。
- **预防措施**：PowerShell 脚本若含中文路径，优先用文件系统 API 动态获取，避免硬编码。

## 五、前端

### FRONTEND-001 Vue 路由中文路径 404

- **发现模块**：通用
- **现象**：生产环境访问含中文的路由路径返回 404，本地开发正常。
- **根因**：Nginx 未配置 `try_files $uri $uri/ /index.html`，History 模式路由刷新后找不到页面。
- **解决方案**：Nginx location 块中加 `try_files $uri $uri/ /index.html;`。
- **预防措施**：部署前参考 `specs/00-conventions/14-部署与运维规范.md § Nginx 配置`。

## 六、AI / LLM

### AI-001 LLM 上下文超长导致截断

- **发现模块**：SPEC-102 会话消息
- **现象**：历史会话条数多时，AI 回复质量下降，出现"忘记"前文信息的情况。
- **根因**：历史消息全量拼入 Prompt，超过模型 Context Window 限制后被截断。
- **解决方案**：实现滑动窗口策略，只保留最近 N 轮对话；超出时使用摘要压缩历史。
- **预防措施**：AI 上下文构建层必须做 Token 计数，超过阈值（如 3000 tokens）时触发压缩。

## 七、性能

### PERF-001 N+1 查询问题

- **发现模块**：通用
- **现象**：列表页加载慢，监控发现单次页面请求触发 50+ SQL 查询。
- **根因**：在循环中调用单条查询接口，未使用批量查询 + 内存聚合。
- **解决方案**：先批量查询 IDs，再 `IN (...)` 一次性拉取关联数据，在 Service 层做 Map 聚合。
- **预防措施**：代码审查中重点检查 `for` 循环内的 DB 操作；集成测试开启 SQL 打印并人工 review。
