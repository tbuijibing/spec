# 测试规范

> 来源：`plans/测试策略文档.md`

---

## 一、测试目标

- 功能符合需求规格。
- 接口、前端、数据库行为一致。
- 每次迭代有回归依据。
- 验收结果能沉淀到版本归档中。

---

## 二、测试层次

| 层次 | 目标占比 | 说明 |
|------|----------|------|
| 单元测试 | 60% | 函数、服务、核心业务逻辑 |
| 集成测试 | 30% | API、数据库、缓存、消息队列协作 |
| E2E / 验收测试 | 10% | 浏览器真实操作、完整业务闭环 |

---

## 三、每模块必测项

| 类别 | 要求 |
|------|------|
| API 测试 | 正常、边界、异常、未授权、越权 |
| 功能测试 | 主流程、关键分支、回退路径 |
| 数据测试 | 建表、写入、更新、软删除、索引命中 |
| 前端测试 | 页面渲染、交互、状态提示、空状态 |
| 安全测试 | 租户隔离、权限、输入攻击 |
| 回归测试 | 修改点关联功能回归 |

---

## 四、验收要求

每个模块上线前必须完成：

1. 浏览器全流程真实操作
2. 截图存档
3. `SPEC-XXX-9-验收报告.md` 填写完成
4. 版本目录 `versions/vX.Y/test-report/` 归档完成

---

## 五、覆盖率目标

| 范围 | 目标 |
|------|------|
| 核心服务层 | ≥80% |
| 高风险模块 | ≥85% |
| 工具类 / 适配层 | 视价值评估，但不得零测试 |

---

## 六、与模块文档关系

| 文档 | 作用 |
|------|------|
| `SPEC-XXX-5-验收文档.md` | 定义测试范围与用例 |
| `SPEC-XXX-9-验收报告.md` | 记录验收结果与结论 |
| `versions/vX.Y/test-report/` | 保存该版本所有测试证据 |

---

## 七、禁止项

- 禁止“代码写完但未验收”就标记完成。
- 禁止只测接口，不测真实页面流程。
- 禁止修改后不做回归。
- 禁止没有测试证据就写“通过”。
